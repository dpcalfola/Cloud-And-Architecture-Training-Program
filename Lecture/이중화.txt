1. 이중화의 목석
    - 시스템의 장애로부터 고가용성을 위해 장비 및 연결을 다중화
    - 장애가 발생하면 준비된 장비를 서비스에 투입 -> 짧은 시간 내에 서비스 복구가 가능
    - 업데이트와 설정 변경시 이중화를 통한 무중단 서비스 가능



    1.1 Fail-over (시스템 대체 기능)
        - 서버 또는 시스템의 장애가 발생할 경우 -> 대체장비(서비스)가 동작 -> 지속성 유지


    1.2 Load-balance (부하분산)
        - 두개 이상의 연결 또는 서비스 사용시 특정 서비스에 부하가 발생 -> 트래픽을 분산신다
        -> 부하를 분산 시킬 수 있다




2. 이중화의 구성(설정) 방법

    2.1 Active - Standby
        - 가장 기본적인 HA(High Availability) 구성방법으로
        - 다중화 장치중 하나의 서비스만 활성화
        - 장애 발생시 대체 장치로 서비스를 이동시켜 Fail-over 기능을 수행
        - Fail-over 시 서비스가 중단되므로 서비스 중단시간이 발생

    2.2 Active - Active
        - 다중화 서비스르 ㄹ모두 활성화 시켜 서비스 분산을 통한 안정성 유지
        - 장애가 발생하여도 서비스 중단 없이 지속적으로 서비스 가능
        - Fail-over 시 리소스 처리량이 증가하여 시스템 장애로 이어질 수 있음으로 리소스 분산 계획 필요

            2.2.1 Hot Standby
                - 모든 장비를 기동중

            2.2.2 Warm Standby
                - 장비의 운영체제는 올라와 있으나 최소한으로만 사용하여 대기하고 있음 (Sync 안맞는 상태)

            2.2.3 Cold Standby
                - 대체 장비가 꺼져 있음 -> 장애 발생시 운영체제를 올리고 Fail-over 수행



3. Fail-over
    3.1
        - 평소 허용하는 장치(서비스)와 그 복제 장치를 가지고 있음
        - 사용장치 장애로 사용이 어려울 경우 복제장치로 그 작업을 이전 처리
        - 무중단 서비스를 구축해주는 것을 의미




4. Stateful / Stateless
    4.1 개념: 상태를 저장하는지 여부에 따라 구분
        - Stateless: 상태를 저장하지 않는 서버
        - Stateful: 상태를 저장하는 서버


    4.2 Stateless
        - 상태를 저장하지 않는 서버
        - 연결 상태를 유지하지 않음(Async)
        - UDP, HTTP 등
        - 복사 개념

    4.3 Stateful
        - 상태를 저장하는 서버
        - 연결 상태 유지(Sync)
        - TCP, FTP 등
        - 복제 개념

    4.4 복제와 복사의 개념 구분
        - 복사: 현제 상태만 기록
        - 복제: 원본과 복제본이 동기화 됨



5. 대칭/비대칭 라우팅
    5.1 대칭형 라우팅
        - 출발지에서 목적지로 가능 경로와 목적지에서 출발지로 돌아오는 경로가 같은 경우
    5.2 비대칭형 라우팅
        - 출발지에서 목적지로 가능 경로와 목적지에서 출발지로 돌아오는 경로가 다른 경우



6. NAT 명렫어 정리
    6.1
        ip nat inside source A B
        -> 인사이드에서 들어오는 패킷의 소스 아이피를 A에서 B로 변환
    6.2
        ip nat outside source A B
        -> 아웃사이드에서 들어오는 패킷의 소스 아이피를 A에서 B로 변환

    6.3 ip nat inside destination
        - outside 에서 inside 로 나가는 패킷에 대해 destination 을 변환
        - 내부 망에 있는 장비들이 로드밸런싱을 가능하게 함
        -> 목적지 주소를 제어하여 트래픽을 분산한다는 뜻


7. 백업 스케줄 고려사항

    7.1 RPO (Recovery Point Objective)
        - 장애 발생시 서비스 연속석을 위해 어느시점으로 백업 할지 결정하는 지표
        - 데-이터 손실을 얼마나 감당 할 수 있는지 RPO 결정 포인트
        - 감내 할 수 있는 데이터 손실의 양(시간)을 의미
        - 현재(장애)로부터 가장 가까운 복원 시점(백업) 까지의 시간 목표

    7.2 RTO (Recovery Time Objective)
        - 목표 목구 시간
        - 장애 발생시 시스템을 원상테로 복원하는데 소요되는 시간
        - 시스템이 얼마나 빠르게 복구 되어야 하는지 결정하는 지표
        - 서비스가 재개될 떄까지 걸리는 시간

    7.3 RIO (Recovery Location Objective)
        - 장애 발생시 데이터 종류벌로 저장 위치에 따라 다앙한 DC(Data Center) 사용
        - (과거에는 자기 테이프를 이용한 저장 ~ 현 클라운드 환경 이용)

    7.4 RLO (Recover Level Objective)
        - 목표 복구 레벨
        - 장애 발생시 복구할 순서를 지정하여 데이터의 우선 순위에 따라 복구 정도를 결정
        - (운영체제, 데이텁제이스, 소스코드....)

    7.5 RSO(Recovery Scope Objective)
        - 비즈니스 중요도에 따라 복구 시스템 선정

    7.6 RCO(Recovery Communication Objective):
        - 목표 네트워크 복구 시간?
        - DR 센터와 주요 거점간 네트워크 복구 수준 및 범위
